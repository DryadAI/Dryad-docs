# Post-mortem · dryad-forge bring-up (2026-05-13 → 2026-05-14)

## Summary
Installing Arch Linux on dryad-forge took 6 failed iterations of a custom offline ISO before we pivoted to the stock upstream Arch ISO + scripted online install over SSH. The pivot succeeded on the first attempt. Total time burnt on the offline ISO path: a full night. **Forge is up; the custom ISO is not the right tool for this job today.**

## What we tried

| # | Approach | Result |
|---|---|---|
| 1 | Custom ISO; installer fires via `dryad-install.service` with `After=network-online.target` | Service never fired — `network-online.target` never reached on a live archiso with no DHCP. Fixed by switching to `After=multi-user.target` and forcing output to `/dev/tty1`. |
| 2 | Same ISO, real boot | Kernel boots, drops to emergency shell: `mount: wrong fs type, bad option, bad superblock on ''`. Empty device. |
| 3 | Patched `hooks/archiso` to include `TYPE=="disk"` in `_search_for_archisodevice` | Verified file present in source overlay. Built ISO had **no archiso hook in the initramfs at all**, only `memdisk` + `udev`. |
| 4 | Added the corresponding **install scripts** (`install/archiso`, `install/archiso_loop_mnt`) to the overlay while keeping `mkinitcpio-archiso` deliberately OUT of `packages.x86_64`; verified patched hook now landed in initramfs via `lsinitcpio -l` | initramfs contained the patched hook. Forge still dropped to emergency shell on real hardware. |
| 5 | Added an autologin chain (`.zlogin` → `.automated_script.sh` with a 5-second countdown to `install.sh`), disabled the systemd service to avoid a race | Forge could not get to a usable console at all — the underlying archiso boot problem from #2 / #4 was not actually fixed. |
| 6 | **Pivot.** Stock upstream Arch ISO → boot → manually set root password, start sshd, rsync our scripts in, run `install.sh` | install.sh got through partition / format / mount cleanly, then `pacstrap -U` failed with dozens of "unable to satisfy dependency" errors because the offline payload was 36 top-level packages, not the full ~240-package closure. |
| 7 | Replaced `pacstrap -U` with `pacstrap -K` (online) — forge had DHCP internet anyway — kept all the rest of the configuration logic | Pacstrap succeeded (240 packages), GRUB installed, reboot into installed system at static 10.10.10.20. |

## Root causes

There were **two independent bugs**, either of which would have killed the offline ISO path on its own.

### Bug 1 — initramfs device search misses whole-disk ISO9660

`mkinitcpio-archiso`'s runtime hook `_search_for_archisodevice` builds its candidate device list with:

```sh
lsblk -no PATH -Q 'RM == 1 && ( TYPE == "part" || TYPE == "rom" )'
```

`mkarchiso` writes the ISO9660 filesystem onto the **whole** USB block device (`TYPE=="disk"`), not into a partition. On HP Envy Slice G2 hardware the kernel enumerates the USB as a disk with no usable partitions for the search, so the loop returns nothing, the eventual `mount` is called with an empty source, and the kernel drops the user to an emergency shell. Patching the predicate to include `TYPE=="disk"` is the right direction, but on this hardware that patch alone did not make the symptom go away — further root-causing was not completed before we pivoted.

### Bug 2 — "offline" payload was the top of the dependency graph, not the closure

`/home/katalyst/dryad-arch-iso/airootfs/root/packages/` contained 36 `.pkg.tar.zst` files: roughly the contents of `packages.x86_64` plus a handful of dryad-specific extras. The real dependency closure of `base base-devel linux linux-firmware grub …` is ~240 packages. `pacstrap -U` will not install a package whose runtime dependencies are not on disk, so the install died midway with a cascade of unsatisfied-dependency errors (xz, pcre2, audit, openssl, glibc, libcap, acl, attr, libxcrypt, ncurses, pam, readline, shadow, systemd-libs, util-linux-libs, …). The bundle would have to be regenerated by walking the full transitive closure (`pacman -Sp` → sync into the airootfs cache) before offline install could succeed.

## Operational gotchas (write these down)

- `rsync -a` against a freshly booted archiso `/root` rewrites it with `uid:gid` from the source host. The resulting ownership trips `sshd`'s `StrictModes` and breaks pubkey login. Use `-rz` (or `-r --no-perms --no-owner --no-group`) when pushing to a live ISO's home dir. Recovery: `chown root:root /root && chmod 750 /root` on the target.
- The archiso airootfs is a **COW tmpfs around ~256 MB**. Anything material — package bundles, keyrings, large rsync transfers — must be staged in `/tmp` (sized to system RAM, ~7.8 GB on forge) and symlinked into `/root` if a `/root/...` path is required.
- `mkinitcpio` `install/*` scripts are the **build-time wrappers** that decide whether a hook is even attempted. Patching the runtime hook (`hooks/foo`) without the install script (`install/foo`) means `mkinitcpio` silently skips the hook entirely. The only reliable verification is `lsinitcpio -l` on the produced initramfs.
- `mkinitcpio-archiso` being absent from `packages.x86_64` is **load-bearing** for any custom hook patches — including it would let `pacstrap` overwrite the patched overlay file during the chroot build. Keep it out of the package list and ship `install/` + `hooks/` via the airootfs overlay.
- `/etc/mkinitcpio.d/linux.preset` is **not owned** by the `linux` package (verify with `pacman -Qo`). Overlay copies of this file survive pacstrap — useful for forcing the live-system `mkinitcpio` to use a custom HOOKS list.
- `post-forge.sh` currently has a silent-skip bug: it logs `[post-forge] Configuring dnsmasq...` but does not actually install or configure the package. dnsmasq must be installed by hand (or the script patched). Tracked as a follow-up to CHG-036.

## What worked, and why

Stock Arch ISO + scripted SSH install worked on the first attempt because:

- The stock ISO is the build path the upstream archiso team actually tests. Whatever hardware-specific behavior was sinking our custom ISO on this HP doesn't reproduce on the stock one.
- DHCP on forge's home-LAN cable gave us internet for free, so we could use online `pacstrap` and stop trying to ship a self-contained closure.
- All the parts of `install.sh` that are not pacstrap — partition / format / `genfstab` / locale / users / networkd / GRUB / SSH key / marker — are environment-independent and ran identically over SSH.

## Recommendation

Adopt "stock ISO + scripted SSH install" as the documented bring-up procedure (see runbook). The custom offline ISO is a worthwhile longer-term project — useful for fully air-gapped sites and for the Phase 7 K8s cluster nodes — but should be revisited only after we have: (a) a script that builds the full transitive package closure into `airootfs/root/packages/`, and (b) a working diagnosis of the whole-disk archiso boot issue on the HP Envy Slice hardware, or a confirmation that the patched hook works on the OptiPlex cluster nodes instead.
